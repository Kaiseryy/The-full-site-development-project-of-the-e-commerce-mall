## 第一章：项目概述

> **本章你会学到**：这个项目是什么、它长什么样、为什么 Google 要做它、以及你学完能收获什么。

### 1.1 这个项目是什么？

**Online Boutique**（在线精品店）——你可以把它理解为"微服务版的淘宝"。

想象一下：你打开一个电商网站，上面有各种商品（墨镜、手表、吹风机……），你可以把它们加入购物车、切换货币看价格、下单、收到确认邮件。整个购物体验和淘宝京东一模一样。

但区别在于：淘宝的背后是一个巨大的系统工程，而这个项目的背后是 **11 个独立的小程序（微服务）在分工协作**。每个小程序只做一件事——有人负责展示网页、有人负责管购物车、有人负责算汇率——它们通过一种叫 gRPC 的高速"电话线"互相沟通。

它是一个**教学项目**，代码是真实可运行的，但支付是模拟的、发货是模拟的、邮件也是模拟的。目的不是上线卖货，而是**教你如何用微服务架构构建真实的云原生应用**。

```
+-----------------------------------------------------------------+
|                     Online Boutique 在线精品店                      |
|                                                                 |
|   墨镜    背心    手表    乐福鞋    吹风机                           |
|   烛台    调料瓶  竹罐    马克杯                                    |
|                                                                 |
|   支持 6 种货币实时切换  |  购物车  |  模拟下单                       |
|                                                                 |
|   背后：11 个微服务 x 5 种编程语言 x Kubernetes 集群                  |
+-----------------------------------------------------------------+
```

### 1.2 业务场景

打开网站后，你可以完整体验一个电商流程：

| 步骤 | 操作 | 背后发生了什么（先睹为快） |
|------|------|------------------------|
| 1 | **打开首页** | frontend 服务向 productcatalogservice 要了 9 个商品数据，向 currencyservice 换算了价格 |
| 2 | **浏览商品详情** | 点击商品后，系统还顺便调用了推荐服务"你可能还喜欢…" |
| 3 | **切换货币** | 从 USD 切到 EUR，页面上的所有价格瞬间变化——currencyservice 在背后疯狂工作 |
| 4 | **加入购物车** | 点击 Add to Cart，商品 ID 和数量被 cartservice 存入了 Redis |
| 5 | **查看购物车** | 系统从 Redis 读取你的购物车，同时调用物流服务估算运费 |
| 6 | **下单结账** | 最复杂的一步——checkoutservice 协调 6 个服务：查购物车→算总价→扣款→发货→清空购物车→发邮件 |
| 7 | **收到确认邮件** | emailservice 发送模拟邮件（实际只打印日志） |

> 关键认知：上面的 7 个步骤，每一个都涉及 3~7 个微服务在背后协同工作。这就是微服务架构的核心魅力——每个服务各司其职，通过协作完成复杂业务。

### 1.3 为什么 Google 要做这个项目？

Google 用它来展示如何用 Google Cloud 的产品（GKE、Cloud Service Mesh、Cloud Operations 等）来现代化企业应用。它是一份"活的广告"和"最好的教材"。

它的核心价值：

| 价值点 | 通俗理解 |
|--------|---------|
| **多语言混合开发** | 11 个服务用了 Go、C#、Node.js、Python、Java 五种语言，就像联合国开会，各国代表说自己最擅长的语言 |
| **gRPC 通信** | 服务之间全部用 gRPC 通信（Google 自家技术），比传统的 HTTP REST 快 7-10 倍 |
| **Kubernetes 原生部署** | 所有服务都跑在 Kubernetes 上，展示了当今最主流的云原生部署方式 |
| **可观测性** | 内置了 Tracing（请求追踪）、Profiling（性能剖析）、Health Check（健康检查）等生产级特性 |
| **开源 & 免费** | 代码完全开源在 GitHub，你可以随便看、随便改、随便部署 |

## 第二章：项目架构总览

> **本章你会学到**：整个系统的全景图——就像看一张城市地图，你会知道每条路通向哪里、每个建筑是干什么的。读完这章，你就能在脑中建立起整个系统的"三维模型"。

### 2.1 先理解两种架构

在深入之前，我们先搞清楚"单体架构"和"微服务架构"的区别。这就像两种不同的餐厅经营方式：

**单体架构 — 就像一个"超级大排档"：**

```
+-------------------------------------------------------------+
|                      单体应用 (Monolith)                       |
|                                                             |
|   +----------+  +----------+  +----------+  +----------+   |
|   | 商品模块  |  | 购物车   |  | 支付模块  |  | 物流模块  |   |
|   +----------+  +----------+  +----------+  +----------+   |
|                                                             |
|   所有功能挤在一个大程序里，一损俱损，牵一发而动全身              |
|   改一行代码 -> 整个系统重新编译、重新部署（可能要几十分钟）       |
+-------------------------------------------------------------+
```

**微服务架构 — 就像"美食广场"：**

```
+-------------------------------------------------------------+
|                    微服务架构 (Microservices)                   |
|                                                             |
|   +----------+  +----------+  +----------+  +----------+   |
|   | 商品摊位  |  | 购物车摊  |  | 支付摊位  |  | 物流摊位  |   |
|   |  (Go)    |  |  (C#)    |  | (Node.js)|  |  (Go)    |   |
|   +----------+  +----------+  +----------+  +----------+   |
|                                                             |
|   每个摊位独立经营、独立升级、独立扩容，互不影响                 |
|   改支付摊位 -> 只重新部署支付摊位（几秒钟），其他摊位照常营业     |
+-------------------------------------------------------------+
```

| 对比维度 | 单体架构（大排档） | 微服务架构（美食广场） |
|---------|------------------|---------------------|
| 修改影响范围 | 改一处要重部署全部 | 只部署改动的服务 |
| 技术选型 | 只能选一种语言 | 每个服务选最合适的语言 |
| 扩容方式 | 整个系统复制一份 | 只扩容繁忙的服务 |
| 调试难度 | 简单（都在一个进程里） | 复杂（要追踪跨服务的调用链） |
| 运维复杂度 | 低 | 高（需要 K8s、服务发现、监控等） |

### 2.3 11 个微服务一览

这 12 个运行单元（11 个微服务 + 1 个 Redis）分工明确，各司其职：

| # | 服务名 | 语言 | 端口 | 一句话职责 | 难度 |
|---|--------|------|------|-----------|------|
| 1 | **frontend** | Go | 8080 | Web 前端，用户的"接待员"。不存数据，只负责调度和渲染 | 两星 |
| 2 | **cartservice** | C# | 7070 | 购物车管家，把购物车数据存在 Redis 里 | 三星 |
| 3 | **productcatalogservice** | Go | 3550 | 商品仓库，从 JSON 文件读 9 个商品数据 | 一星 |
| 4 | **currencyservice** | Node.js | 7000 | 汇率计算器，使用欧洲央行真实汇率数据，全系统最忙的服务 | 两星 |
| 5 | **paymentservice** | Node.js | 50051 | 模拟收银台，假装扣款并返回交易 ID | 一星 |
| 6 | **shippingservice** | Go | 50051 | 模拟物流，估算运费 + 给出追踪号 | 一星 |
| 7 | **emailservice** | Python | 5000 | 模拟邮件发送，用 Jinja2 模板渲染确认邮件 | 一星 |
| 8 | **checkoutservice** | Go | 5050 | **下单总导演**，协调 6 个服务完成下单流程 | 四星 |
| 9 | **recommendationservice** | Python | 8080 | 推荐引擎，"你可能还喜欢…" | 两星 |
| 10 | **adservice** | Java | 9555 | 广告投放，根据商品类别返回匹配广告 | 两星 |
| 11 | **loadgenerator** | Python/Locust | - | 压力测试机器人，持续模拟用户浏览/加购/下单 | 一星 |
| 额外 | **redis-cart** | Redis | 6379 | 购物车数据的"临时记事本"，内存存储、极快 | 一星 |

### 2.4 服务之间的调用关系

```
                        frontend (Go)
                       "中央调度台"
            +------+------+------+------+------+
            |      |      |      |      |      |
            v      v      v      v      v      v
        product  curren  cart  recom  ship   checkout
        catalog   cy     svc   mend    ping   svc
          svc     svc    (C#)  (Py)   (Go)   (Go)
         (Go)   (Node)   |                      |
                  |      |                      |
                  |      v                      |
                  |   [Redis]                   |
                  |                      +------+------+------+
                  |                      |      |      |      |
                  |                      v      v      v      v
                  |                   payment shipping email  cart
                  |                     svc     svc     svc    svc
                  |                   (Node)   (Go)   (Py)   (C#)
                  |                      |      |      |      |
                  |                      +------+------+      |
                  |                        checkout 下单时调用   |
                  |                                            |
                  +--------------------------------------------+
                          frontend 浏览时也会直接调用
```

**关键观察**：
- `frontend` 是用户流量的唯一入口，自己不处理业务，只负责调度
- `checkoutservice` 是下单流程的**总导演**（Orchestrator），把支付、发货、邮件串成一个完整流程
- 多数服务只依赖 0~1 个其他服务，耦合度很低——这就是好的微服务设计
- Redis 只被 cartservice 使用，隔离清晰

### 2.5 完整请求流程：从浏览到下单

下面这张图展示了用户从打开首页到完成下单的全过程。**建议你对着这张图反复看，直到能把每一步在脑海中模拟出来：**

```
用户打开首页 (GET /)
    |
    +-> frontend
    |   +-> (1) gRPC -> productcatalogservice.ListProducts()      -> 9 个商品数据
    |   +-> (2) gRPC -> currencyservice.GetSupportedCurrencies()  -> 货币列表
    |   +-> (3) gRPC -> cartservice.GetCart(sessionID)            -> 购物车
    |   +-> (4) gRPC -> currencyservice.Convert() x 9次           -> 转换每个商品价格
    |   +-> 渲染 HTML -> 返回首页
    |
用户点击某个商品 (GET /product/OLJCESPC7Z)
    |
    +-> frontend
    |   +-> (5) gRPC -> productcatalogservice.GetProduct(id)      -> 商品详情
    |   +-> (6) gRPC -> recommendationservice.ListRecommendations -> "你可能还喜欢"
    |   +-> (7) gRPC -> adservice.GetAds(categories)              -> 相关广告
    |   +-> 渲染 product.html
    |
用户点击 "Add to Cart" (POST /cart)
    |
    +-> frontend
    |   +-> (8) gRPC -> cartservice.AddItem(sessionID, productID, qty)
    |       +-> Redis: SET sessionID -> Cart序列化字节
    |
用户进入购物车页面 (GET /cart)
    |
    +-> frontend
    |   +-> (9)  gRPC -> cartservice.GetCart(sessionID)           -> 购物车内容
    |   +-> (10) gRPC -> productcatalogservice.GetProduct() x N   -> 每件商品信息
    |   +-> (11) gRPC -> currencyservice.Convert() x N            -> 价格转换
    |   +-> (12) gRPC -> shippingservice.GetQuote()               -> 估算运费
    |   +-> 渲染 cart.html
    |
用户点击 "Place Order"（下单！）(POST /cart/checkout)
    |
    +-> frontend
    |   +-> (13) gRPC -> checkoutservice.PlaceOrder(...)
    |       |
    |       +-> (14) gRPC -> cartservice.GetCart()              -> 获取购物车
    |       +-> (15) gRPC -> productcatalogservice.GetProduct() -> 获取价格
    |       +-> (16) gRPC -> currencyservice.Convert()          -> 货币转换
    |       +-> (17) gRPC -> shippingservice.GetQuote()         -> 估算运费
    |       +-> (18) gRPC -> paymentservice.Charge(amount)      -> 扣款 -> TXN_ID
    |       +-> (19) gRPC -> shippingservice.ShipOrder()        -> 发货 -> TRACK_ID
    |       +-> (20) gRPC -> cartservice.EmptyCart()            -> 清空购物车
    |       +-> (21) gRPC -> emailservice.SendConfirmation()    -> 发确认邮件
    |
    +-> frontend 渲染 order.html -> 订单确认页面
```

> 上面的编号 (1)~(21) 代表了 21 次 gRPC 调用。在一个完整的"浏览→下单"流程中，**至少发生 21 次服务间调用**。这就是微服务架构的现实：一次用户操作背后是无数次服务协作。

### 第二章小结

- 微服务架构就像**美食广场**，每个摊位独立经营，互不影响
- 本项目有 11 个微服务，frontend 是入口，checkoutservice 是下单总导演
- 用户浏览首页时至少触发 4 次 gRPC 调用，下单时 checkoutservice 协调 6 个服务
- 服务之间耦合度低，大多数服务只依赖 0~1 个其他服务

**思考题**：如果 frontend 服务挂了，用户还能正常购物吗？为什么？（答案：不能，因为它是唯一面向用户的服务，所有流量都要经过它）

**handlers.go — 首页处理器（展示完整的数据聚合过程）：**

```go
func (fe *frontendServer) homeHandler(w http.ResponseWriter, r *http.Request) {
    // === 第1步：获取支持的货币列表 ===
    // 比如 ["USD", "EUR", "CAD", "JPY", "GBP", "TRY"]
    currencies, err := fe.getCurrencies(r.Context())
    // 如果出错，给用户显示错误页面
    if err != nil { /* 显示错误 */ }

    // === 第2步：获取所有商品 ===
    // 从 productcatalogservice 拿到 9 个商品的完整数据
    products, err := fe.getProducts(r.Context())
    if err != nil { /* 显示错误 */ }

    // === 第3步：获取当前用户的购物车 ===
    // 从 Redis（通过 cartservice）读取购物车数据
    cart, err := fe.getCart(r.Context(), sessionID(r))
    if err != nil { /* 显示错误 */ }

    // === 第4步：货币转换（循环每个商品）===
    // 把每个商品的美元价格转换成用户选择的货币
    for i, p := range products {
        price, err := fe.convertCurrency(
            r.Context(), 
            p.GetPriceUsd(),           // 商品的美元价格
            currentCurrency(r),        // 用户选择的货币（如 "EUR"）
        )
        ps[i] = productView{p, price}  // 组合：商品 + 转换后的价格
    }

    // === 第5步：渲染 HTML 并返回给浏览器 ===
    // 把上面收集的所有数据塞进 home.html 模板
    err = templates.ExecuteTemplate(w, "home", map[string]interface{}{
        "products":   ps,             // 商品列表（含转换后价格）
        "cart_size":  cartSize(cart), // 购物车商品数量
        "currencies": currencies,     // 货币列表
        "ad":         fe.chooseAd(...), // 广告
    })
}
```

> 这段代码告诉我们的重要事实：frontend 的 `homeHandler` 方法在返回 HTML 之前，至少调用了 3 个后端服务。如果其中一个服务挂了，首页就渲染不出来。这就是微服务的"短板效应"。

#### 核心代码解读

**RedisCartStore.cs — 添加商品到购物车：**

```csharp
public async Task AddItemAsync(string userId, string productId, int quantity)
{
    // ===== 第1步：从 Redis 读取当前购物车 =====
    // userId 就是 Redis 的 Key，比如 "session-abc123"
    var value = await _cache.GetAsync(userId);
    // _cache 是 Redis 连接，GetAsync 就像 Redis 的 GET 命令

    Hipstershop.Cart cart;  // Protobuf 定义的 Cart 类型
    if (value == null)
    {
        // ==== 第2a步：用户还没有购物车 -> 新建 ====
        cart = new Hipstershop.Cart();
        cart.UserId = userId;
        cart.Items.Add(new Hipstershop.CartItem {
            ProductId = productId,
            Quantity = quantity
        });
    }
    else
    {
        // ==== 第2b步：用户已有购物车 -> 反序列化后更新 ====
        // 把 Redis 中的二进制数据还原成 Cart 对象
        cart = Hipstershop.Cart.Parser.ParseFrom(value);
        
        // 查找购物车中是否已有这个商品
        var existingItem = cart.Items.SingleOrDefault(i => i.ProductId == productId);
        
        if (existingItem == null)
            // 新商品：追加到列表
            cart.Items.Add(new Hipstershop.CartItem { /* ... */ });
        else
            // 已有商品：累加数量（而不是创建重复条目）
            existingItem.Quantity += quantity;
    }

    // ===== 第3步：序列化购物车并写回 Redis =====
    // ToByteArray() 把 Cart 对象变成 Protobuf 二进制（比 JSON 小很多）
    await _cache.SetAsync(userId, cart.ToByteArray());
}
```

**设计亮点**：
- 使用**接口 `ICartStore`** 抽象存储层，可以轻松切换 Redis -> Spanner -> AlloyDB
- 购物车数据用 **Protobuf 序列化**存入 Redis，效率比 JSON 高 3-5 倍
- 同一商品多次加入，自动**累加数量**而非创建重复条目

### 3.4 currencyservice — 货币转换服务

| 属性 | 值 |
|------|---|
| **语言** | Node.js |
| **代码路径** | `src/currencyservice/` |
| **端口** | 7000 |
| **协议** | gRPC |
| **角色** | 汇率计算器 |

#### 通俗理解

这是全系统**最忙的服务**。它的工作很简单：把一种货币的金额转换成另一种货币。但因为首页 9 个商品、每个都要转换价格，购物车页又要转换，所以它的调用量是其他服务的 10 倍以上。

```
首页渲染一次 = 调用 currencyservice 至少 9+1 = 10 次
（9个商品转换 + 1次获取货币列表）
```

**有趣的细节**：它使用的汇率数据来自**欧洲央行（ECB）的真实汇率**，所以转换结果是真实可信的（不是瞎编的）。

#### gRPC 接口 — Money 的巧妙设计

```protobuf
message Money {
    string currency_code = 1;  // 货币代码，如 "USD"
    int64 units = 2;           // 整数部分，如 19（表示 19 美元）
    int32 nanos = 3;           // 纳秒（十亿分之一）部分
                               // 如 990000000 表示 0.99 美元
}
```

**为什么 Money 用 `units + nanos` 而不是 `float`（浮点数）？**

因为计算机的浮点数无法精确表示金额。比如 `0.1 + 0.2 = 0.30000000000000004`（不等于 0.3），这在金融系统中是不可接受的。

Google 的解决方案：把金额拆成"整数部分 + 十亿分之一部分"，全部用整数运算。整数运算永远是精确的！

```
$19.99
= units: 19         （整数部分：19 美元）
+ nanos: 990000000  （纳秒部分：0.99 美元 = 990,000,000 x 10^(-9) 美元）
= 精确的 $19.99，没有任何浮点误差
```

#### 通俗理解

Step 2: 获取购物车 --------> cartservice.GetCart()

Step 4: 货币转换 ----------> currencyservice.Convert() x N

Step 6: 计算总金额 --------> total = 运费 + 求和(商品价格 x 数量)

Step 8: 发货！-------------> shippingservice.ShipOrder() -> trackID

#### 核心代码解读

```go
func (cs *checkoutService) PlaceOrder(ctx context.Context, 
    req *pb.PlaceOrderRequest) (*pb.PlaceOrderResponse, error) {
    
    // (1) 生成唯一订单 ID（UUID 保证全局唯一）
    orderID, _ := uuid.NewUUID()

    // (2) 准备订单：一口气做完"获取购物车->查价格->转换货币->估算运费"
    prep, _ := cs.prepareOrderItemsAndShippingQuoteFromCart(
        ctx, req.UserId, req.UserCurrency, req.Address)

    // (3) 计算总金额 = 运费 + 商品价格x数量
    total := money.Sum(shippingCost, itemPrice1 * qty1, itemPrice2 * qty2, /* ... */)

    // (4) 扣款！这是最关键的一步，一旦失败整个订单就失败
    txID, _ := cs.chargeCard(ctx, &total, req.CreditCard)

    // (5) 发货！只有扣款成功后才会发货
    shippingTrackingID, _ := cs.shipOrder(ctx, req.Address, prep.cartItems)

    // (6) 清空购物车（下单成功后购物车应该清空）
    cs.emptyUserCart(ctx, req.UserId)

    // (7) 发送确认邮件（非关键步骤，失败了也不影响订单）
    cs.sendOrderConfirmation(ctx, req.Email, orderResult)

    // (8) 返回完整的订单结果
    return &pb.PlaceOrderResponse{Order: orderResult}, nil
}
```

> 注意：这里没有错误回滚机制！如果第(5)步发货失败，第(4)步的扣款不会退回。生产环境中需要 Saga 模式或分布式事务来处理这种情况。

**设计要点**：
- 典型的 **Saga 编排模式**：checkoutservice 是编排者，负责按顺序调用各服务
- 同步调用链：每步必须等上一步完成才能继续
- 无状态：checkoutservice 自己不存储任何订单数据

### 4.1 gRPC — 微服务通信的"电话线"

#### 先理解问题：微服务之间怎么"说话"？

你有 11 个独立的小程序，它们分布在不同的机器上，甚至用不同的编程语言写的。Go 写的 frontend 怎么告诉 C# 写的 cartservice "帮我加个商品到购物车"？

答案有两种主流方式：
1. **REST API**：HTTP + JSON，就像寄明信片——写地址、贴邮票、投递、等回信
2. **gRPC**：Google 的方案，就像打电话——拨通后直接对话，又快又直接

#### REST vs gRPC 形象对比

```
REST API（寄明信片）                gRPC（打电话）
-----------------                -------------
每次通信都要：                      一次拨通后：
  把数据写成 JSON 文本               建立长连接
  通过 HTTP 发送                    直接对话
  等待 HTTP 响应                    二进制传输
  解析 JSON 文本                    Protobuf 序列化

速度：较慢                          速度：快 7-10 倍
```

| 对比维度 | REST | gRPC |
|---------|------|------|
| 数据格式 | JSON（文本，人类可读） | Protocol Buffers（二进制，机器可读） |
| 传输协议 | HTTP/1.1 | HTTP/2（多路复用、头部压缩） |
| 接口定义 | 无强制标准（Swagger 可选） | `.proto` 文件强制定义 |
| 代码生成 | 手动或第三方工具 | 一行命令自动生成客户端+服务端 |
| 性能 | 较慢（文本解析开销大） | 快（二进制 + HTTP/2） |
| 浏览器支持 | 原生支持 | 需要 grpc-web 代理 |
| 适用场景 | 对外 API、浏览器直接调用 | **微服务内部通信** |

#### gRPC 通信流程（可视化）

```
+--------------------------------------------------------------+
|                     gRPC 通信全流程                            |
|                                                              |
|  客户端 (Go)                       服务端 (Node.js)            |
|  +--------------+                 +--------------+            |
|  | 业务代码      |                 | 业务代码      |            |
|  | Charge(card) |                 | Charge(impl) |            |
|  +------+-------+                 +------+-------+            |
|         |                                |                    |
|         v                                |                    |
|  +--------------+                 +--------------+            |
|  | gRPC Stub    |                 | gRPC Server  |            |
|  | (自动生成)    |                 | (自动生成)    |            |
|  +------+-------+                 +------+-------+            |
|         |                                |                    |
|         v                                |                    |
|  +--------------+                 +--------------+            |
|  | 序列化        |                 | 反序列化      |            |
|  | Protobuf编码  | ---- 网络 ---> | Protobuf解码  |            |
|  +--------------+                 +--------------+            |
|                                                              |
|  整个过程对业务开发者透明——就像调用本地函数一样                   |
+--------------------------------------------------------------+
```

### 4.2 Protocol Buffers — 跨语言的数据"世界语"

#### 一句话理解

Protobuf 就是"比 JSON 更小更快的跨语言数据格式"。它就像联合国的"同声传译"：不管你说什么语言（Go/C#/Java/Python/Node.js），只要你按 proto 定义的格式说话，对方就能准确理解。

#### 本项目 proto 文件解析

所有服务接口定义都在一个文件 `protos/demo.proto` 中——这是整个项目的"宪法"。

```protobuf
// 定义一个"购物车商品"的数据结构
message CartItem {
    string product_id = 1;  // 商品 ID（= 1 是字段编号，不是值！）
    int32  quantity = 2;    // 数量
}

// 定义一个"购物车"
message Cart {
    string user_id = 1;           // 用户 ID
    repeated CartItem items = 2;  // 商品列表（repeated = 数组）
}
```

**字段编号（1, 2, 3...）是 Protobuf 的核心魔法**：它不存字段名，只存编号。所以：

- JSON 存储：`{"product_id": "ABC", "quantity": 3}` -> 约 40 字节
- Protobuf 二进制存储：大概 8 字节

**体积缩小了 5 倍！** 这就是为什么 gRPC 比 REST 快得多。

#### 从 proto 生成代码

```bash
# Go 代码生成（一行命令！）
protoc --go_out=. --go-grpc_out=. protos/demo.proto

# Python 代码生成
python -m grpc_tools.protoc -Iprotos --python_out=. --grpc_python_out=. protos/demo.proto
```

#### 本项目 Dockerfile 分析（多阶段构建的巧妙之处）

```dockerfile
# ========== 第一阶段：编译（重量级） ==========
FROM golang:1.23.4-alpine AS builder   # Go SDK 镜像，大约 300MB
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download                    # 下载所有 Go 依赖
COPY . .
RUN go build -o /go/bin/frontend .     # 编译成单个二进制文件

# ========== 第二阶段：运行（轻量级） ==========
FROM scratch                           # scratch 是空镜像！0 MB！
COPY --from=builder /go/bin/frontend /src/server  # 只复制编译好的二进制
COPY ./templates ./templates            # 复制 HTML 模板
COPY ./static ./static                  # 复制 CSS/图片
EXPOSE 8080
ENTRYPOINT ["/src/server"]             # 启动！
```

**为什么用两阶段构建？**

第一阶段（builder）：有 Go 编译器、源代码、依赖 -> 用于编译。最终产物是一个二进制文件。

第二阶段（scratch）：只复制二进制文件 + 静态资源，不包含编译器、源代码、依赖。

结果：**最终镜像约 20MB**（vs 不用多阶段构建的 300MB+）。镜像越小，启动越快，攻击面越小。

### 4.5 Redis — 购物车的"临时记事本"

#### 一句话理解

Redis 是一个**极快的内存数据库**。普通数据库（如 MySQL）把数据存硬盘，Redis 把数据存内存——所以读写快 100 倍以上。

```
MySQL（硬盘数据库）：  读写 ~1ms      <-- 像从书架上拿书
Redis（内存数据库）：   读写 ~0.01ms   <-- 像从脑海中回忆
```

#### 为什么购物车用 Redis 而不是 MySQL？

| 原因 | 解释 |
|------|------|
| 购物车是临时数据 | 用户可能随时放弃购买，一个月后还有价值的购物车很少 |
| 读写极快 | Redis 是内存数据库，响应时间微秒级别 |
| 可设自动过期 | Redis 可以设置 TTL（Time To Live），自动清理不活跃购物车 |
| 数据结构简单 | 购物车结构就是 Key-Value，天然适合 Redis |

#### 数据存储方式

```
Redis 中的一条购物车记录：
+---------------------------------------------+
| Key:   "session-abc123-def456"               |  <-- 用户 Session ID
| Value: <二进制字节流>                         |     Cart 对象的 Protobuf 序列化
|       解码后 = {                              |
|         user_id: "session-abc123",            |
|         items: [                              |
|           {product_id:"OLJCESPC7Z", qty:2},  |
|           {product_id:"66VCHSJNUP", qty:1}   |
|         ]                                     |
|       }                                       |
+---------------------------------------------+
```

### 4.6 多语言混合开发 — 优势与挑战

#### 为什么用 5 种语言而不是 1 种？

想象一下：如果联合国规定所有人只能说英语，虽然沟通方便，但中文母语者、法语母语者都会很难受。微服务也一样——**用最适合的语言写最合适的服务**。

| 语言 | 本项目中的角色 | 为什么选它 |
|------|-------------|-----------|
| **Go** | frontend / checkout / productcatalog / shipping | 高并发 Web 服务之王，编译快、性能好、部署简单（单个二进制文件） |
| **C#** | cartservice | LINQ 语法写购物车逻辑非常优雅，强类型保证数据安全 |
| **Node.js** | currencyservice / paymentservice | 处理大量 I/O 密集型请求（汇率计算、支付模拟） |
| **Python** | emailservice / recommendationservice | 快速开发、Jinja2 模板渲染邮件、数据分析友好 |
| **Java** | adservice | 企业级生态成熟，适合复杂业务逻辑 |

#### 优势与挑战

| 优势 | 挑战 | 本项目如何应对 |
|------|------|-------------|
| 选最合适的语言 | 通信协议要统一 | 全部用 gRPC + Protobuf，proto 是唯一的"真相来源" |
| 团队各自独立 | 构建流程变复杂 | Skaffold 统一管理 11 个服务的构建 |
| 技术多样性学习 | 运维成本高 | K8s 统一编排，语言不重要，部署方式都一样 |
| 独立升级部署 | 代码规范不统一 | 每个服务独立 repo 结构，互不干扰 |

### 第四章小结

- gRPC 是微服务通信的首选方案，比 REST 快 7-10 倍
- Protobuf 是 gRPC 的数据格式，比 JSON 小 3-10 倍
- Kubernetes 是容器编排平台，帮你管理 Pod/Deployment/Service
- Docker 把应用打包成标准化镜像，一次构建到处运行
- Redis 是内存数据库，适合购物车这类临时、高频读写的数据

**思考题**：如果让你把 currencyservice 从 Node.js 改成 Go 重写，需要修改哪些部分？对其他服务有影响吗？（答案：只需要修改 currencyservice 自己的代码，因为它通过 gRPC 暴露的接口（proto 定义）不变，其他服务无感知！这就是微服务独立部署的优势。）

### 5.6 常见问题排查

## 第六章：项目代码导读

> **本章你会学到**：像侦探一样追踪代码执行路径——从用户点击鼠标到代码第几行执行、调了哪个服务、数据怎么流转。读完这章，你会看懂开源项目的代码结构。

### 6.1 场景一：用户浏览商品列表

**用户操作**：打开浏览器，访问 `http://localhost:8080`

**完整代码追踪**（建议对着源码一起看）：

```
第一步：浏览器发出 HTTP GET / 请求
    |
    v
第二步：frontend/main.go — 路由注册
    r.HandleFunc("/", svc.homeHandler)
    意思：当有人访问 "/" 路径时，调用 homeHandler 函数处理
    
    |
    v
第三步：frontend/handlers.go — homeHandler() 函数
    （下面按执行顺序编号）
    |
    +-> (1) getCurrencies(ctx)
    |    +-> 跳到 frontend/rpc.go -> getCurrencies()
    |        |
    |        | 创建 gRPC 客户端，连接到 currencyservice:7000
    |        | pb.NewCurrencyServiceClient(fe.currencySvcConn)
    |        |   .GetSupportedCurrencies(ctx, &pb.Empty{})
    |        |
    |        | 等价于：对着 currencyservice 喊："有哪些货币可用？"
    |        | currencyservice 回答：["USD","EUR","CAD","JPY","GBP","TRY"]
    |        +-> 返回货币列表
    |
    +-> (2) getProducts(ctx)
    |    +-> 跳到 frontend/rpc.go -> getProducts()
    |        |
    |        | pb.NewProductCatalogServiceClient(fe.productCatalogSvcConn)
    |        |   .ListProducts(ctx, &pb.Empty{})
    |        |
    |        | 等价于：对着 productcatalogservice 喊："把所有商品列出来！"
    |        | productcatalogservice：翻开 products.json，
    |        |   返回 9 个商品（墨镜、背心……）
    |        +-> 返回商品列表
    |
    +-> (3) getCart(ctx, sessionID(r))
    |    +-> 跳到 frontend/rpc.go -> getCart()
    |        |
    |        | pb.NewCartServiceClient(fe.cartSvcConn)
    |        |   .GetCart(ctx, &pb.GetCartRequest{UserId: userID})
    |        |
    |        | 等价于：对着 cartservice 喊：
    |        |   "用户 abc123 的购物车里有什么？"
    |        | cartservice -> Redis: GET abc123 -> 返回购物车数据
    |        +-> 返回购物车
    |
    +-> (4) 循环每个商品，货币转换
    |    for i, p := range products {
    |        convertCurrency(USD价格 -> 用户选择的货币)
    |    }
    |    |
    |    | 对每个商品调用一次 currencyservice.Convert()
    |    | 9 个商品 = 9 次 gRPC 调用！
    |    |
    |    | currencyservice：查汇率表 USD->用户货币，
    |    |   用 units+nanos 精确计算（不用 float！）
    |    +-> 每个商品的价格都转换成了用户选择的货币
    |
    +-> (5) 渲染 HTML
         templates.ExecuteTemplate(w, "home", data)
         |
         | 把前面收集的所有数据（商品列表、购物车大小、货币列表、广告）
         | 填入 home.html 模板 -> 生成完整 HTML

```

**在这个流程中你学到什么？**
- frontend 自己不存任何数据，所有数据都是"实时"从后端服务查询的
- 渲染一个首页需要至少 12 次 gRPC 调用（1 次货币列表 + 1 次商品列表 + 1 次购物车 + 9 次货币转换）
- `convertCurrency` 被循环调用 9 次——这就是为什么 currencyservice 是全系统最忙的服务

**用户操作**：在购物车页面填写地址和信用卡，点击 "Place Order"。

**完整代码追踪**：

```
第一步：浏览器发出 HTTP POST /cart/checkout
    携带：email、address、credit_card_number……
    |
    v
第二步：frontend/handlers.go -> placeOrderHandler()
    |
    | 解析表单数据：email, address, creditCard
    |
    +-> 调用 checkoutservice.PlaceOrder()
        |
        | 把所有信息打包成 PlaceOrderRequest
        | 通过 gRPC 发送给 checkoutservice:5050
        |
        v
第三步：checkoutservice/main.go -> PlaceOrder() — 核心流程！
    |
    +-> (1) 生成订单 ID
    |    orderID = uuid.NewUUID()  // 全局唯一的订单号
    |
    +-> (2) prepareOrderItemsAndShippingQuoteFromCart()
    |    |   （准备订单：一口气做完以下 4 件事）
    |    |
    |    +-> cartservice.GetCart(userID)
    |    |    从 Redis 读取：{items: [{OLJCESPC7Z: 2}, {66VCHSJNUP: 1}]}
    |    |
    |    +-> productcatalogservice.GetProduct(OLJCESPC7Z)
    |    |    返回：Sunglasses, $19.99
    |    |    productcatalogservice.GetProduct(66VCHSJNUP)
    |    |    返回：Watch, $109.99
    |    |

    |    |    $19.99 -> 18.50欧元

    |    |
    |    +-> shippingservice.GetQuote(address, items)
    |         返回：运费 5.00欧元
    |
    +-> (3) 计算总金额
    |    total = 18.50欧元 x 2 + 101.85欧元 x 1 + 5.00欧元 = 143.85欧元
    |
    +-> (4) 扣款！
    |    paymentservice.Charge(143.85欧元, creditCard)
    |    |
    |    | paymentservice 假装验证信用卡、假装扣款
    |    +-> 返回 transactionID = "abc-123-def-456"
    |
    +-> (5) 发货！
    |    shippingservice.ShipOrder(address, items)
    |    |
    |    | shippingservice 假装发货
    |    +-> 返回 trackingID = "TRACK-789-012"
    |
    +-> (6) 清空购物车
    |    cartservice.EmptyCart(userID)
    |    |
    |    | cartservice -> Redis: DEL userID
    |    +-> 购物车清空
    |
    +-> (7) 发送确认邮件
    |    emailservice.SendOrderConfirmation(email, orderResult)
    |    |
    |    | emailservice 用 Jinja2 渲染邮件模板
    |    | （实际只打日志，不真发邮件）
    |    +-> 完成
    |
    +-> (8) 返回 PlaceOrderResponse
         {
           order_id: "...",
           transaction_id: "abc-123-def-456",
           tracking_id: "TRACK-789-012",
           items: [...],
           total: 143.85欧元
         }

第四步：前端渲染
    frontend 收到 PlaceOrderResponse
    +-> 填入 order.html 模板 -> 展示订单确认页

```

**在这个流程中你学到什么？**
- checkoutservice 是完美的**编排服务**案例——它自己不处理业务，只负责按顺序调度

- 从 HTTP 入口 -> gRPC 客户端 -> 服务端实现，三步定位代码
- frontend 是数据聚合者，checkoutservice 是流程编排者
- 复杂流程 = 简单服务的顺序组合

#### 练习 5：编写健康检查脚本（高级）

**目标**：写脚本一次性检查所有 11 个服务是否健康。

```bash
#!/bin/bash
# check-health.sh — 一键体检
SERVICES=(
  "frontend:8080" "cartservice:7070" "productcatalogservice:3550"
  "currencyservice:7000" "paymentservice:50051" "shippingservice:50051"
  "emailservice:5000" "checkoutservice:5050" "recommendationservice:8080"
  "adservice:9555"
)
for svc in "${SERVICES[@]}"; do
  name="${svc%%:*}"; port="${svc##*:}"
  kubectl port-forward "deployment/$name" "$port:$port" &>/dev/null &
  sleep 1
  grpcurl -plaintext "localhost:$port" grpc.health.v1.Health/Check 2>/dev/null \
    && echo "OK $name" || echo "FAIL $name"
  kill %1 2>/dev/null
done
```

### 8.3 常见术语表

| 术语 | 一句话通俗解释 | 类比 |
|------|-------------|------|
| **微服务 (Microservice)** | 把大应用拆成多个独立小应用 | 大超市 -> 美食广场的独立摊位 |
| **单体应用 (Monolith)** | 所有功能混在一个大程序里 | 超级大排档，改一道菜要关整个店 |
| **gRPC** | Google 的高性能远程调用框架 | 服务间的"电话线"，比寄明信片（REST）快 |
| **Protobuf** | gRPC 的二进制数据格式 | 比 JSON 小 3-10 倍的跨语言"世界语" |
| **Kubernetes (K8s)** | 容器的自动管理平台 | 工厂的总控系统，管理所有机器 |
| **Pod** | K8s 最小管理单元 | 工厂里的一个工位 |
| **Deployment** | 管理 Pod 的控制器 | 工头，保证工位上一直有人 |
| **Service** | Pod 的固定网络入口 | 总机号码，不管人怎么换号码不变 |
| **Docker** | 应用打包工具 | 标准集装箱，哪里都能跑 |
| **Dockerfile** | Docker 镜像的"配方" | 集装箱的打包说明书 |
| **Redis** | 极快的内存数据库 | 大脑记忆（vs MySQL 硬盘 = 书本） |
| **Skaffold** | 自动构建+部署工具 | 一键部署流水线 |
| **编排 (Orchestration)** | 协调多个服务完成复杂流程 | 电影导演调度演员、灯光、音效 |
| **负载均衡** | 请求平均分配 | 超市开多个收银台，顾客分散排队 |
| **多阶段构建** | Docker 优化技巧 | 先在大车间组装，成品搬到小车间，大车间扔掉 |
